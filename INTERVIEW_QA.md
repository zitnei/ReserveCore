## 1. 二重予約防止（一番の推しポイント）

### Q.「二重予約はどうやって防いでいますか？」

**A.** 2段構えで防いでいます。

1. **アプリ層の事前チェック**：予約作成前に、同じスタッフ・重なる時間帯の有効な予約がないかDBを検索し、あれば `409 Conflict` を返す（`ReservationService#create`）
2. **DBの EXCLUDE 制約**：それでも**同時アクセス**（ほぼ同時に2件来る）だとアプリのチェックをすり抜けるので、PostgreSQL の `EXCLUDE` 制約で**DB自身が重複を拒否**する（最終防壁）

### Q.「なぜアプリのチェックだけじゃダメなんですか？」

**A.** **競合状態（レースコンディション）**があるからです。
2人がほぼ同時に予約すると、両方とも「事前チェック時点では重複なし」と判定されて、2件とも登録されてしまう。
アプリのチェックは「チェック」と「登録」の間に隙間があるので、同時実行に弱い。
**最終的な整合性はDBに保証させるべき**という考えで、EXCLUDE制約を置いています。

### Q.「EXCLUDE制約って具体的にどう書いたんですか？」

**A.**（`V3__add_exclude_constraint.sql`）
```sql
CREATE EXTENSION IF NOT EXISTS btree_gist;
ALTER TABLE reservations
  ADD CONSTRAINT no_overlap_per_staff
  EXCLUDE USING gist (
      staff_id WITH =,                          -- 同じスタッフで
      tsrange(start_time, end_time, '[)') WITH &&  -- 時間帯が重なったら拒否
  )
  WHERE (status IN ('PENDING', 'CONFIRMED'));   -- 枠を占有する状態だけ対象
```
- `tsrange(..., '[)')` は**半開区間**。`11:00終了`と`11:00開始`は重ならない扱い（連続予約を許可）
- `btree_gist` 拡張は、`staff_id`（=比較）と範囲（&&比較）を1つのGiSTインデックスで併用するために必要
- `WHERE` 句で、キャンセル済みなどは制約対象外（枠を塞がない）

### Q.「DB制約違反はどうハンドリングしていますか？」

**A.** `saveAndFlush` で即座に制約を評価させ、違反時の `DataIntegrityViolationException` を捕捉して、API としては分かりやすい `409 Conflict` に変換しています（`ReservationService#create` の try-catch）。
DBの生々しいエラーをそのままクライアントに返さない、という意図もあります。

## 2. 認証（JWT）

### Q.「認証はどう実装していますか？」

**A.** JWT によるステートレス認証です。
- ログイン成功時にサーバーが JWT を発行
- クライアントは以降 `Authorization: Bearer <token>` を付けてリクエスト
- サーバーは毎回 `JwtFilter` でトークンを検証し、有効なら認証情報を `SecurityContextHolder` にセット
- パスワードは **BCrypt** でハッシュ化して保存（平文では絶対持たない）

### Q.「なぜセッションじゃなくJWTにしたんですか？」

**A.** ステートレスにできるからです。
- セッションはサーバー側に状態を持つ（スケールアウト時に共有が必要）
- JWT はトークン自体に情報が入っているので、**サーバーが状態を持たずに済む**
- 今回はSPA（Next.js）とAPIを分離した構成なので、ステートレスなJWTが相性が良い

### Q.「JWTのデメリットは？どう対処しますか？」

**A.** **発行後すぐ無効化できない**のが弱点です（有効期限が切れるまで有効）。
対策としては、有効期限を短くしてリフレッシュトークンを併用するのが定石。
今回は学習用なので有効期限24時間のシンプルな実装にしていますが、
**本番ならリフレッシュトークンの導入が必要**だと認識しています（READMEのロードマップにも記載）。

## 3. 認可（権限制御）

### Q.「権限管理はどうしていますか？」

**A.** **3層**で多層防御しています。

1. **URLレベル**（`SecurityConfig`）：`/api/auth/**` は公開、それ以外は認証必須
2. **メソッドレベル**（`@PreAuthorize("hasRole('ADMIN')")`）：店舗登録やユーザー一覧はADMINのみ
3. **ドメインレベル**（サービス層）：「自店のスタッフか」「予約の本人か」を判定して `403`

### Q.「なぜ3層も必要なんですか？1つじゃダメ？」

**A.** **層ごとに守れる範囲が違う**からです。
- URL/メソッドレベルは「ロール」までしか見れない（ADMINかSTAFFか）
- でも「**STAFFは自分の店舗だけ操作可能**」のような**データに依存した権限**は、
  実際のデータを見ないと判定できない → サービス層でやる必要がある
- 「画面で制御するから大丈夫」ではなく、**バックエンド単体で安全**にするのが目的

### Q.「具体的に『自店判定』ってどこでやってますか？」

**A.** `storeStaffRepository.existsByStore_IdAndUser_Id(...)` で、
「このユーザーがこの店舗のスタッフとして登録されているか」を中間テーブルで確認しています。
予約作成時も「指定スタッフがその店舗に所属しているか」をこれで検証しています。

## 4. スキーマ管理（Flyway）

### Q.「DBのスキーマ管理は？」

**A.** **Flyway** でマイグレーション管理しています（`V1`〜`V3`）。
- `ddl-auto: validate` にして、**エンティティとDBスキーマの乖離を起動時に検知**
- Hibernateの自動DDL生成（`update`）は本番で危険なので使わない
- スキーマの変更履歴がSQLファイルとしてGit管理される＝**いつ誰が何を変えたか追える**

### Q.「なぜ ddl-auto を update にしないんですか？」

**A.** `update` は予測不能だからです。
Hibernateが勝手にカラムを足したりするが、**消す方向の変更はできない**し、本番で意図しない変更が走るリスクがある。
Flywayで明示的にSQLを書く方が、**何が起きるか完全にコントロールできる**。

## 5. テスト

### Q.「テストはどう書きましたか？」

**A.** 実PostgreSQLに対する**結合テストを46件**書きました。
- MockMvc で実際のHTTPリクエスト〜レスポンスを検証
- 権限（401/403）、バリデーション（400）、二重予約（409）、DB制約まで検証
- 共通基盤 `IntegrationTestSupport` で、各テスト前にFK整合を保つ順序で全テーブルを初期化
  → **テストの実行順に依存しない**設計

### Q.「なぜ単体テストじゃなく結合テスト中心？」

**A.** このアプリの肝が「**認可とDB制約**」だからです。
モックを多用した単体テストだと、実際のDB制約（EXCLUDE）や Spring Security の挙動が検証できない。
**実DBに対して通しで動くことを保証したかった**ので結合テスト中心にしました。

### Q.「CIは？」

**A.** GitHub Actions で、push/PRごとに**実PostgreSQLコンテナを立てて全46テストを自動実行**しています。
READMEのバッジで常にグリーンが見えるようにしています。

## 6. 設計・アーキテクチャ

### Q.「全体の構成は？」

**A.** レイヤードアーキテクチャです。
- `api`（コントローラ + リクエスト/レスポンスDTO）
- `domain`（エンティティ + リポジトリ + サービス＝業務ロジック）
- `security`（JWT・フィルタ・Security設定）
- `common`（例外・統一エラーレスポンス）
- DTOは Java の `record`、エンティティとレスポンスを分離し、パスワード等の機密はAPIに露出させない

### Q.「エラーハンドリングは？」

**A.** `@RestControllerAdvice`（`GlobalExceptionHandler`）で例外を一元処理し、
`{status, message, fieldErrors, timestamp}` の**統一フォーマット**で返します。
バリデーションエラーはフィールド単位のメッセージも返す。
ステータスは 400/401/403/404/409/500 を適切に使い分け。

## 7. デプロイ（聞かれたら）

### Q.「どうやって公開してるんですか？」

**A.**
- バックエンド + DB：**Render**（Dockerでデプロイ、`render.yaml` でDB込みのワンクリック構成）
- フロントエンド：**Vercel**
- 環境変数でCORS許可Originやシークレットを注入

### Q.「デプロイで詰まったことは？」（←正直に話すと好印象）

**A.** 2つありました。
1. Render の DB接続文字列が `postgresql://...` 形式で、JDBCが要求する `jdbc:postgresql://...` と違って起動失敗 → host/port/dbを個別取得してURLを組み立てて解決
2. APIとDBのリージョンがずれて内部接続できなかった → 同一リージョンに統一
→ **本番環境特有のハマりどころを自分で調べて解決**した経験になりました。
